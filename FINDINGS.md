# triage-bench Findings

## Score Progression

| Run | Prompt | Accuracy | Bug F1 | Feature F1 | Question F1 |
|-----|--------|----------|--------|------------|-------------|
| run_001 | baseline_v1 | 70.4% | 0.746 | 0.862 | 0.396 |
| run_002 | baseline_v2 | 75.0% | 0.754 | 0.835 | 0.671 |
| run_003 | baseline_v3_apo | 71.0% | 0.749 | 0.841 | 0.485 |

Model: `claude-haiku-4-5-20251001`. Test set: 200 stratified examples from NLBSE 2024 (67 bug / 67 feature / 66 question).

---

## Headline

The handwritten v2 prompt outperformed both the zero-shot baseline (+4.6pp) and MIPROv2's optimized v3 (+4.0pp). The harness made this visible. Without it, v3 would have looked like a reasonable next step — longer, more structured, generated by an optimizer — and would have shipped as a regression. The value of this project is not the v2 prompt; it's the measurement layer that caught v3.

---

## What moved the score

### 1. The question class was the entire game (v1 → v2, +4.6pp overall)

v1's zero-shot prompt had no definition distinguishing bugs from questions. The model defaulted to `bug` whenever it was uncertain — which meant 40 of 66 actual questions were predicted as bugs (question recall = 27.7%). v2 added:

- An explicit definition requiring bugs to have both unexpected behavior AND an expected-vs-actual contrast
- A hard rule that technical content (stack traces, version strings, error messages) does not make something a bug
- Two few-shot examples where the title was maximally bug-shaped (a linker warning verbatim; a Python exception verbatim) but the label was `question`

That single change took question recall from 27.7% to 72.7% and question F1 from 0.396 to 0.671 — accounting for the full +4.6pp overall gain.

### 2. The few-shot examples were load-bearing, not decorative

Removing them (as MIPROv2 did in v3) caused question recall to collapse back to 37.9% — close to the v1 baseline of 27.7%. The APO-generated instruction is longer and more structured than v2's, but without examples that concretely show the bug/question boundary on hard cases, the model reverts to surface-pattern matching: error message in title → bug.

### 3. There is a stable bug/feature boundary; the bug/question boundary is the hard problem

`feature` was never the problem class. Both v1 and v2 achieved F1 > 0.83 on `feature` without any special handling. The entire precision-recall space being traded is between `bug` and `question`. Every prompt that improves question recall does so by sacrificing some bug precision — the model is shifting mass from `bug` to `question`. v2 found a better point on that tradeoff curve than v1 or v3.

---

## What didn't move the score

### 1. APO instruction length and structure (v3)

MIPROv2 produced a 250-word structured instruction with bullet-point checklists for each class, a meta-rule about linguistic features vs domain knowledge, and explicit "look for" lists. Despite being objectively more detailed than v2, it scored lower. The generated instruction includes "Look for: Explicit error messages or stack traces" under **Bug** — which directly re-introduces the surface-signal bias that v2's definitions were designed to suppress. More words ≠ better signal.

### 2. MIPROv2's zero-demo choice

MIPROv2 selected zero few-shot demonstrations for the final optimized program, relying entirely on the improved instruction. This may reflect optimization noise: the train set had ~10 rate-limit errors during evaluation (skipped examples), and the train distribution had 34/33/33 vs the test set's 67/67/66 — enough to make the optimization landscape noisy. The optimizer's decision to drop examples was the most consequential single choice, and it hurt.

### 3. Elaborating the `feature` definition

Neither v2 nor v3 changed how `feature` is defined relative to v1, and `feature` performance was stable across all three runs (F1 0.835–0.862). Adding detail there would have been wasted effort.

### 4. What APO surfaced (but couldn't act on alone)

MIPROv2 did produce one genuinely non-obvious observation worth carrying forward: "The patterns of these categories are consistent across different frameworks and domains." This makes explicit what v2 only implied — don't over-index on domain vocabulary, focus on linguistic and structural signals. But that insight, embedded in an instruction that dropped the few-shot examples, wasn't enough to overcome the boundary confusion it was meant to address. The meta-rule is useful; the configuration that produced it wasn't.

---

## What to try next (if continuing)

1. **Hybrid v4:** Take v2's definitions + hard rules + the two hard few-shot examples, and add MIPROv2's "domain-agnostic" meta-rule. Run it through the same eval.
2. **Re-run APO with a fairer config:** The open question is whether v3 measured MIPROv2's true ceiling or its default settings. The optimizer ran with ~10 skipped examples due to rate-limiting, a 34/33/33 train distribution that doesn't match the test set, and a zero-demo outcome that may reflect noise rather than a genuine preference. Before concluding APO doesn't help here, run it with rate-limit noise eliminated and demos forced on (`max_labeled_demos=4`).
3. **Fix the rate-limit noise in optimization:** Reduce `num_threads=1` and add a sleep to respect the 50 RPM ceiling during DSPy's eval loop, so no examples are skipped during optimization.
4. **Expand to routing and priority:** Now that classification is measured and iterable, add team routing as a second label. The measurement infrastructure (runner, scorer, compare) is already reusable.
